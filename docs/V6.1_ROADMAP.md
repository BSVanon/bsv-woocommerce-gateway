### Implementation Required (4-6 hours)

#### 1. Payment Receiver Endpoint
**File:** `includes/bip270-payment-receiver.php` (new)

**Functionality:**
- Register endpoint: `add_action('woocommerce_api_bsv_payment', 'BWWC__receive_bip270_payment')`
- Accept POST with `Content-Type: application/payment`
- Validate request (order_id, key, signature)
- Parse raw transaction hex
- Verify transaction outputs match PaymentTerms
- Broadcast transaction to BSV network
- Return PaymentACK response
- Update order state with txid

**Key Functions:**
```php
BWWC__receive_bip270_payment()           // Main endpoint handler
BWWC__parse_bsv_transaction($raw_hex)    // Extract outputs from raw tx
BWWC__verify_payment_outputs($tx, $expected) // Validate outputs match
BWWC__broadcast_transaction($raw_hex)    // Broadcast via WhatsOnChain/Bitails
BWWC__generate_payment_ack($txid, $memo) // Build PaymentACK response
BWWC__check_payment_idempotency($order_id, $txid) // Prevent double-processing
```

#### 2. Transaction Parsing
**Challenge:** BSV transactions are binary format

**Approach:**
- Decode raw hex to binary
- Parse version (4 bytes)
- Parse inputs (varint count + input data)
- Parse outputs (varint count + output data)
- Extract: amount (8 bytes LE) + script length (varint) + script
- Verify against expected outputs from PaymentTerms

**Libraries to Consider:**
- Pure PHP implementation (no dependencies)
- Or use existing BSV transaction parser if available

#### 3. Transaction Broadcasting
**Endpoints:**
- Primary: `https://api.whatsonchain.com/v1/bsv/main/tx/raw`
- Fallback: `https://api.bitails.io/tx/broadcast`

**Error Handling:**
- Network errors → retry with fallback
- Already in mempool → treat as success (idempotent)
- Invalid transaction → return 400 with error details
- Broadcast timeout → return 503

#### 4. Idempotency Protection
**Storage:**
- Meta key: `_bip270_submitted_txids` (array of txids)
- Check if txid already processed
- If duplicate: return cached PaymentACK (don't rebroadcast)

#### 5. PaymentACK Response
**Format (JSON):**
```json
{
  "payment": "<original-payment-hex>",
  "memo": "Payment received for Order #123",
  "merchantData": {
    "orderId": 123,
    "txid": "abc123..."
  }
}
```

**Headers:**
- `Content-Type: application/payment-ack`
- `Cache-Control: no-cache`

## Priority 3: BRC-100 postMessage Security Hardening

### Current State (v6.0.0)
- ⚠️ XDM uses `targetOrigin='*'`
- ⚠️ No `event.source` validation
- ⚠️ No `event.origin` validation
- ✅ Random request ID prevents replay
- ✅ `isTrusted` flag checked
- ✅ 5-second timeout

### Hardening Required
**File:** `assets/js/bsv-brc100-payment.js` (lines 210-244)

**Changes:**
1. Store expected window reference when sending
2. Validate `event.source === expectedWindow`
3. Optionally validate `event.origin` against allowlist
4. Use specific `targetOrigin` instead of `'*'` when possible

**Example:**
```javascript
cwiXdmCall: function(call, params = {}, target = window.parent, origin = '*') {
    return new Promise((resolve, reject) => {
        const id = Math.random().toString(16).slice(2) + Date.now().toString(16);
        
        const onMessage = (e) => {
            if (!e.isTrusted) return;
            
            // NEW: Validate source window
            if (e.source !== target) return;
            
            // NEW: Optionally validate origin
            // if (origin !== '*' && e.origin !== origin) return;
            
            const d = e.data || {};
            if (d.type !== 'CWI' || d.isInvocation || d.id !== id) return;
            
            // ... rest of handler
        };
        
        // ... rest of function
    });
}
```

**Estimated Effort:** 1-2 hours
